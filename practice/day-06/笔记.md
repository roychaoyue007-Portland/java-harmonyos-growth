## 阶段 2：核心关系，30 分钟

今天要先记住这条关系：

```text
客户端：与用户交互、主动发起请求的一方
服务端：接收请求、处理数据、返回结果的一方
IP：帮助客户端找到网络中的哪一台机器
端口：帮助客户端找到这台机器上的哪一个具体服务
HTTP：客户端和服务端之间共同遵守的通信规则
```

客户端不是凭空找到服务端的。客户端通常先根据网址或地址找到服务端所在的机器，再通过端口找到这台机器上的具体服务，最后按照 HTTP 这种共同规则发送请求和接收响应。

## 阶段 3：动手观察，60 分钟

在 PowerShell 中轻量观察自己的网络地址和一次 HTTP 请求：

```powershell
ipconfig
curl.exe -I https://example.com
```

观察重点：

- `ipconfig` 中的 IPv4 地址表示当前电脑在当前网络里的地址。
- `curl.exe -I https://example.com` 会向 example.com 发起一次 HTTP 请求并查看响应头。
- HTTP 状态码可以帮助判断服务端是否成功处理了请求。
- 这次观察可以说明：客户端发请求，服务端返回响应。

如果出现报错，可以从这些方向排查：

1. 如果 `curl.exe` 无法访问，可能是网络、代理或 DNS 解析问题。
2. 如果找不到命令，可能是当前终端环境不识别该命令。
3. 如果返回非 200 状态码，先观察状态码含义，不急着深入排查。
4. 今天只建立客户端、服务端、IP、端口、HTTP 的整体关系，不展开网络底层细节。

## 我的原始验收答案

1. 与用户交互的一方，发送请求的一方；如鸿蒙某 App、浏览器客户端。
2. 处理请求方数据的一方，一般不直接对接用户；主要负责处理请求等。
3. 用来表明服务端在哪个地址，客户端可以通过 IP 地址找到服务端在网络上的地址。
4. 端口主要用来区分具体服务。
5. 是一种标准通信协议；在两者之间主要是为了保证双方能听懂对方的消息。
6. 客户端通过 IP 地址 + 端口找到服务端，发送请求给服务端，使用 HTTP 协议保障交流无障碍。
7. 服务器不止一个功能，所以需要运行多个服务，端口是专门用来找到指定服务的。
8. 鸿蒙 App 是客户端，Java 后端是服务端，HTTP 是两者之间无障碍交流的通信协议。

## 我的验收答案整理

1. 客户端是与用户交互、主动发起请求的一方。常见例子有浏览器、鸿蒙 App、手机 App、桌面软件。
2. 服务端是接收客户端请求、处理数据并返回结果的一方。它通常在后台运行，不直接面对普通用户，主要负责处理业务逻辑、读写数据、返回响应结果。
3. IP 地址用来标识网络中的一台设备。客户端可以通过服务端的 IP 地址找到服务端所在的机器。它主要解决“去哪台机器”的问题。
4. 端口用来区分一台机器上的不同服务。只知道 IP 只能找到机器，端口可以继续定位到这台机器上的具体服务。
5. HTTP 是客户端和服务端之间常用的一种通信协议。它规定客户端如何发送请求、服务端如何返回响应，让双方按照共同规则交流。
6. 浏览器作为客户端，会先根据网址找到服务端地址。网址通常会被解析成 IP 地址，然后浏览器连接服务端对应端口，按照 HTTP 或 HTTPS 协议发送请求。服务端接收请求并处理，再返回响应，浏览器收到响应后展示结果。
7. 一台服务器本质上是一台机器，它可以同时运行多个程序或服务。不同服务需要用不同端口区分。IP 负责找到这台机器，端口负责找到这台机器上的具体服务。
8. 鸿蒙 App 是客户端，Java 后端是服务端，HTTP 是客户端和服务端之间通信的协议。鸿蒙 App 通过 HTTP 请求 Java 后端，Java 后端处理后通过 HTTP 返回结果。

## 补充理解

今天最需要记住的是：客户端通过 IP 找到服务器这台机器，通过端口找到机器上的具体服务，再用 HTTP 这种共同规则完成请求与响应。浏览器输入网址时，中间通常还有“域名解析成 IP”的过程，但今天只需要先知道它会从网址找到服务端地址。
